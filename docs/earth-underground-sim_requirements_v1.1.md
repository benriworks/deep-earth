---

## 1. 目的・概要

地球の内部構造を、科学的に正確かつ視覚的に直感的・インタラクティブに体験できる Web アプリケーションを構築する。

本アプリケーションでは、Three.js を中心とした 3D 表現を用いて、地球内部の層構造、地震波の伝播、マントル対流、仮想プローブによる探索などを体験できることを目指す。

---

## 2. 想定ユーザーとユースケース

本セクションは v1.0 と同一とする。

想定される主な利用シーンは以下の通り。

- 教育現場における地球科学・地学の視覚教材としての利用
- 学習者が地球内部構造を直感的に理解するためのインタラクティブ教材としての利用
- 地震波やマントル対流など、直接観察しにくい現象を可視化するシミュレーションとしての利用
- 科学データをもとにした 3D ビジュアライゼーションのデモアプリケーションとしての利用

## 4. 非機能要件

v1.1 では、非機能要件を Vercel デプロイおよびローカル Node.js 開発前提に合わせて更新する。

### 4.1 パフォーマンス要件

| 項目 | 要件 |
|---|---|
| フレームレート | 現代ブラウザで 60FPS 以上を安定して維持すること |
| 3D 最適化 | Next.js + Three.js の最適化を最大限活用すること |
| 推奨実装 | React Three Fiber の利用を推奨すること |
| 初回ロード時間 | 本番ビルド時に 4 秒以内を目標とすること |

### 4.2 開発環境要件

| 項目 | 要件 |
|---|---|
| Node.js | Node.js 20 以上を必須とする |
| 言語 | TypeScript を使用する |
| TypeScript 設定 | strict mode を有効にする |
| 推奨フレームワーク | Next.js 15（App Router） |
| パッケージマネージャー | pnpm または bun を推奨する |
| 開発体験 | ホットリロードを利用できること |
| 静的解析 | TypeScript 厳格チェック、ESLint、Prettier を導入すること |
| Storybook | 任意導入とする |

### 4.3 デプロイ環境要件

| 項目 | 要件 |
|---|---|
| 本番デプロイ先 | Vercel とする |
| 利用プラン | 無料プランで十分動作する構成を想定する |
| 自動デプロイ | GitHub 連携による自動デプロイを想定する |
| 分析 | Vercel Analytics / Speed Insights の活用を推奨する |
| API 拡張 | 将来的に Serverless Functions による軽量 API を利用可能とする |

### 4.4 ユーザビリティ・アクセシビリティ要件

本セクションは v1.0 と同一とする。

- レスポンシブデザインに対応すること。
- PC、タブレット、スマートフォンで基本的な閲覧・操作が可能であること。
- WCAG 相当のアクセシビリティを考慮すること。
- 視覚情報に依存しすぎず、必要に応じてラベル、説明文、補助 UI を提供すること。

### 4.5 科学性・正確性要件

本セクションは v1.0 と同一とする。

- 地球内部構造に関する科学的に妥当なデータを使用すること。
- レイヤー構造、深度、密度、温度などのデータは PREM モデル近似値などを参考にすること。
- 教育用途で誤解を招かない表現とすること。
- 3D 表現上の簡略化や誇張がある場合は、UI 上で説明すること。

## 5. 技術要件

### 5.1 推奨技術スタック

| 分類 | 技術 | 理由・コメント |
|---|---|---|
| フレームワーク | Next.js 15（App Router） | Vercel に最適化されており、TypeScript 標準対応、ルーティングが容易 |
| 3D レンダリング | Three.js + `@react-three/fiber` + `@react-three/drei` | React と相性が良く、開発効率が高い |
| UI | Tailwind CSS + shadcn/ui | モダンで美しい UI を高速に構築できる |
| 状態管理 | Zustand または Jotai | 軽量で Three.js との相性が良い |
| 型定義 | TypeScript（strict mode） | 大規模開発・保守性を重視できる |
| アニメーション | framer-motion + Three.js | UI アニメーションと 3D 表現を両立できる |
| データ管理 | JSON / TypeScript 型定義 | レイヤーデータを静的に管理しやすい |
| テスト | Vitest + React Testing Library | 将来的な追加を推奨する |
| デプロイ | Vercel | ワンクリックデプロイ、プレビュー環境を利用できる |

```bash

```

## 6. データ要件

本セクションは v1.0 と同一とする。

### 6.1 使用データ

- PREM モデル近似値などを参考にした地球内部構造データを使用する。
- 地殻、マントル、外核、内核などの各層について、深度、半径、密度、温度などを定義する。
- 初期リリースでは、外部 API ではなく静的データとして管理する。

### 6.2 データ管理方針

| 項目 | 方針 |
|---|---|
| データ形式 | JSON または TypeScript オブジェクト |
| 管理場所 | `lib/earthData.ts` など |
| 更新方法 | コード管理下で更新する |
| 外部 API | 初期リリースでは不要 |

## 7. 制約条件

### 7.1 開発・デプロイ制約

| 項目 | 制約 |
|---|---|
| Artifacts 環境 | Claude Artifacts の制限は一切受けない |
| 開発環境 | ローカル Node.js 環境で開発する |
| 実装方式 | クライアントサイド中心で実装する |
| 3D 実装 | Three.js + React を中心に構成する |
| API 利用 | 必要に応じて Vercel Serverless Functions を軽く使用可能とする |
| 将来的な API 例 | ユーザー生成地震データの保存など |
| Vercel 制約 | Vercel 無料プランで動作する範囲に収める |
| リソース制約 | 帯域・コンピュートリソースを過度に消費しない設計とする |

### 7.2 パフォーマンス制約

- Three.js の複雑なシミュレーションは、パフォーマンスを最優先に設計する。
- 重い計算は Web Workers や `requestIdleCallback` の活用を検討する。
- 3D オブジェクト数、ポリゴン数、テクスチャサイズを適切に制御する。
- 不要な再レンダリングを避けるため、React コンポーネントの分割とメモ化を適切に行う。

## 10. 修正ポイントまとめ

| 項目 | v1.0（Artifacts版） | v1.1（Vercel版） | 変更理由 |
|---|---|---|---|
| 実行環境 | Claude Artifacts | ローカル Node.js → Vercel | 要望に合わせた変更 |
| フレームワーク | 純粋 HTML / JS | Next.js 15 | 開発効率・型安全性・デプロイ性の向上 |
| 3D ライブラリ | Three.js | Three.js + `@react-three/fiber` | React との親和性向上 |
| 状態管理 | なし | Zustand | 大規模化対応 |
| デプロイ | Artifacts 共有リンク | Vercel | 要望に合わせた変更 |
| 拡張性 | 低い | 高い | Functions 追加が容易で将来性が高い |
